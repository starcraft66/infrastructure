== Overview

[.text-justify]
I currently operate two personal kubernetes clusters, `k8s-235` and `k8s-305-1700`, each at a different physical site. Both clusters are deployed on bare metal servers running NixOS and are provisioned by the Nix expressions and modules located in the `nixos` folder at the root of the repo. The kubernetes cluster components are deployed using a custom NixOS module that I wrote specific to my infrastructure. Most importantly, it handles end-to-end mutual TLS certification of all cluster components using Hashicorp Vault and vault-agent.

[.text-justify]
The `k8s-235` cluster consists of 3 consolidated control-plane, worker and etcd nodes (`soarin`, `sassaflash`, `stormfeather`). These three nodes also run Hashicorp Vault (in HA mode with Raft storage), Patroni (HA PostgreSQL) and Pocket ID (OIDC identity provider), all behind keepalived VIPs. These services along with etcd are hosted directly on NixOS rather than inside the kubernetes cluster because the cluster itself depends on them for PKI, secrets, authentication and database access, and running them in-cluster would create chicken-and-egg dependency problems. The `k8s-305-1700` cluster is a single-node cluster running on `spike`, featuring an NVIDIA GPU for GPU-accelerated workloads and Multus CNI for host network bridging. Like the 235 cluster, Vault and etcd run directly on NixOS on `spike` for the same dependency reasons.

== Provisioning

[.text-justify]
The physical servers providing compute for the kubernetes nodes are provisioned by Nix. The flake in this repo contains NixOS configurations for the physical servers `soarin`, `sassaflash` and `stormfeather` (k8s-235) as well as `spike` (k8s-305-1700). Deployments are performed remotely using https://colmena.cli.rs/[Colmena] (e.g. `colmena apply --on=@k8s-235`).

.Procedure to install a new physical NixOS server
[%collapsible]
====
. Create the host configuration directory `nixos/hosts/<hostname>/` with `default.nix`, `configuration.nix`, `hardware.nix`, `bootloader.nix`, `keys.nix` and any role-specific modules (e.g. `kubernetes.nix`). Register the host in `nixos/hosts/default.nix` in the `systems` attrset.

. Build a customized install ISO for the host. The resulting ISO includes the host's network configuration, allowing remote SSH access to the installer.
+
[source,console]
----
nix build .#packages.x86_64-linux.<hostname>
----

. Boot the target server from the ISO (via USB or IPMI/iLO virtual media). The live environment will come up with the host's static IP and SSH configuration active.

. Partition the disks manually. The layout depends on the host's storage requirements. Refer to existing `hardware.nix` files for examples.
+
.LUKS + Btrfs (used by the 235 cluster nodes)
[source,console]
----
# Partition the disk (create EFI/boot and root partitions)
cryptsetup luksFormat /dev/<root-partition>
cryptsetup open /dev/<root-partition> cryptroot
mkfs.btrfs /dev/mapper/cryptroot
mount /dev/mapper/cryptroot /mnt
btrfs subvolume create /mnt/root
btrfs subvolume create /mnt/nix
umount /mnt
mount -o subvol=root /dev/mapper/cryptroot /mnt
mkdir /mnt/nix
mount -o subvol=nix /dev/mapper/cryptroot /mnt/nix
mkfs.ext4 /dev/<boot-partition>
mkdir /mnt/boot
mount /dev/<boot-partition> /mnt/boot
----
+
.ZFS (used by spike, lava)
[source,console]
----
zpool create -o ashift=12 -O mountpoint=none <poolname> /dev/<disk>
zfs create -o mountpoint=legacy <poolname>/root
zfs create -o mountpoint=legacy <poolname>/nix
zfs create -o mountpoint=legacy <poolname>/var
zfs create -o mountpoint=legacy <poolname>/home
mount -t zfs <poolname>/root /mnt
mkdir -p /mnt/{nix,var,home,boot}
mount -t zfs <poolname>/nix /mnt/nix
mount -t zfs <poolname>/var /mnt/var
mount -t zfs <poolname>/home /mnt/home
mount /dev/<efi-partition> /mnt/boot
----

. Generate initrd SSH host keys for FDE remote unlock.
+
[source,console]
----
# For LUKS+Btrfs hosts (235 cluster pattern)
mkdir -p /mnt/etc/secrets/initrd
ssh-keygen -t rsa -N "" -f /mnt/etc/secrets/initrd/ssh_host_rsa_key
ssh-keygen -t ed25519 -N "" -f /mnt/etc/secrets/initrd/ssh_host_ed25519_key

# For ZFS hosts (spike, lava pattern)
ssh-keygen -t ed25519 -N "" -f /mnt/boot/initrd-ssh-key
----

. Install NixOS to the target disk using the flake. You will need a copy of or access to the infrastructure repo from the installer environment.
+
[source,console]
----
nixos-install --flake <path-to-repo>#<hostname>
----

. Reboot the server into the installed system. Unlock FDE remotely using the appropriate just recipe.
+
[source,console]
----
just unlock-235    # or just unlock-305-1700
----

. Provision Vault agent credentials on the new host. Follow the procedure in `vault/terraform/vault-agent.adoc` to create role IDs and secret IDs for all vault-agent instances on the host (kubernetes, etcd, coredns, front-proxy).

. Perform the first Colmena deployment with secrets. Set `deploy-keys = true` in the host's `keys.nix` to push Vault TLS certificates, Patroni credentials and other secrets, then deploy.
+
[source,console]
----
colmena apply --on=<hostname>
----
+
After the initial deployment succeeds, revert `deploy-keys` to `false` to avoid excessive YubiKey interactions on routine deploys.

. (If using sops-nix for on-host secret decryption) Convert the host's SSH key to a PGP key, add it to `.sops.yaml` and re-key the relevant secrets.
+
[source,console]
----
ssh-to-pgp -i /etc/ssh/ssh_host_rsa_key -o secrets/keys/hosts/<hostname>.asc
# Note the fingerprint, add it to .sops.yaml, then re-key
----
====

.Procedure to initialize, backup and restore the etcd database
[%collapsible]
====
The etcd data directory is `/var/lib/etcd`. TLS certificates are managed by vault-agent and stored in `/var/lib/secrets/etcd/`. The etcd cluster is used by both the kubernetes API server and Patroni (on k8s-235), so operations on etcd affect both services.

===== Initializing a new cluster

Set `initialClusterState = "new"` in the cluster's NixOS module (e.g. `nixos/modules/k8s-235-1/default.nix`), then deploy with Colmena. Once the cluster is bootstrapped and healthy, revert to `initialClusterState = "existing"` and deploy again.

===== Taking a snapshot

Run on any etcd member node:

[source,console]
----
etcdctl snapshot save /tmp/etcd-snapshot.db \
  --endpoints=https://$(hostname -f):2379 \
  --cacert=/var/lib/secrets/etcd/etcd-ca.pem \
  --cert=/var/lib/secrets/etcd/server.pem \
  --key=/var/lib/secrets/etcd/server-key.pem
----

===== Restoring from a snapshot

IMPORTANT: This is a disruptive operation. Restoring etcd will affect both kubernetes state and Patroni's DCS state (leader election, cluster configuration). Coordinate accordingly.

. Stop all dependent services on every node in the cluster.
+
[source,console]
----
# On each node:
systemctl stop kube-apiserver patroni etcd
----

. Remove the old data directory on each node.
+
[source,console]
----
rm -rf /var/lib/etcd/member
----

. Restore the snapshot on each node, specifying the correct node name and peer URLs. Repeat for each member, adjusting `--name` and `--initial-advertise-peer-urls` accordingly.
+
[source,console]
----
etcdctl snapshot restore /tmp/etcd-snapshot.db \
  --name <hostname> \
  --data-dir /var/lib/etcd \
  --initial-cluster "soarin=https://soarin.235.tdude.co:2380,stormfeather=https://stormfeather.235.tdude.co:2380,sassaflash=https://sassaflash.235.tdude.co:2380" \
  --initial-advertise-peer-urls https://<hostname>.235.tdude.co:2380
----

. Restart services on each node in order.
+
[source,console]
----
systemctl start etcd
# Verify cluster health:
etcdctl endpoint health \
  --endpoints=https://$(hostname -f):2379 \
  --cacert=/var/lib/secrets/etcd/etcd-ca.pem \
  --cert=/var/lib/secrets/etcd/server.pem \
  --key=/var/lib/secrets/etcd/server-key.pem
systemctl start kube-apiserver
systemctl start patroni
----

== Components

[.text-justify]
The core components of the kubernetes clusters include the control-plane itself, etcd, the Cilium CNI networking plugin and the democratic-csi CSI storage plugin. Cilium is installed at cluster provisioning time via the NixOS kubernetes module and democratic-csi is later installed by ArgoCD. Cilium can then be further upgraded by ArgoCD.

=== 🐝 Cilium

[.text-justify]
Both clusters use https://cilium.io/[Cilium] as their CNI networking plugin, providing eBPF-based networking with native routing (no overlays or tunnels between nodes). Cilium replaces kube-proxy entirely and handles service load-balancing using DSR (Direct Server Return). It is configured for dual-stack (IPv4 + IPv6) operation on both clusters with BGP peering to advertise cluster pod and service subnets to the wider network. The k8s-235 cluster peers with the core Brocade switch and with AS208914 routers, while k8s-305-1700 peers with the 305-1700 site gateway. https://docs.cilium.io/en/stable/observability/hubble/[Hubble] is enabled for network observability on both clusters.

=== 🐙 ArgoCD

[.text-justify]
ArgoCD is a central part of my clusters. Once the initial control-plane for a cluster is deployed by NixOS, the only application that must be installed manually into the cluster is ArgoCD. Once installed, it will take over the lifecycle of all of the applications in this repository to be deployed into the cluster.
ArgoCD deploys everything in the `k8s/apps` folder including itself. A commit webhook in GitLab will cause ArgoCD to sync the repository on every commit. Authentication to ArgoCD is done through Hashicorp Vault's OIDC provider.

[.text-justify]
To deploy ArgoCD on the k8s-235 cluster, go into `k8s/apps/argocd` and run `kustomize build --enable-alpha-plugins overlays/235-prod | kubectl apply -f -`. For k8s-305-1700, use `overlays/305-1700-prod` instead. ArgoCD will become available at https://gitops.235.tdude.co (k8s-235) or https://gitops.305-1700.tdude.co (k8s-305-1700).

==== Kustomize

[.text-justify]
Kustomize touts itself to be a template-free solution to manage application deployments on kubernetes in stark contrast to the popular Helm package manager. All of the manifests in `k8s/apps` are assembled with kustomize before being applied. ArgoCD runs kustomize which builds the manifests and applies them to the cluster.

==== SOPS

ArgoCD is capable of decrypting secrets in this repo through the https://github.com/viaduct-ai/kustomize-sops[ksops] plugin for kustomize. To do this, an `initContainer` installs the `ksops` binary into ArgoCD's repo-server Pod and a second `initContainer` imports a GPG private key ArgoCD uses to decrypt secrets. Secret generators in this repo use the `ksops-exec` kind from `viaduct.ai/v1`. All sops-encrypted secrets you expect ArgoCD to be able to decrypt must be encrypted with ArgoCD's public key.

.Procedure to generate a GPG key for use with ArgoCD
[%collapsible]
====
ArgoCD needs a passphrase-less GPG key to decrypt SOPS secrets non-interactively. The private key is stored as a SOPS-encrypted Kubernetes Secret (`deploy-pgp-key`) in each ArgoCD overlay, and the fingerprint must be listed in `.sops.yaml` so that all k8s secrets are encrypted with it.

. Generate a new passphrase-less GPG key.
+
[source,console]
----
gpg --batch --gen-key <<EOF
%no-protection
Key-Type: RSA
Key-Length: 4096
Name-Real: ArgoCD SOPS
Name-Email: argocd@tdude.co
Expire-Date: 0
%commit
EOF
----
+
Note the fingerprint from the output.

. Export the private key in ASCII armor.
+
[source,console]
----
gpg --armor --export-secret-keys <fingerprint> > argocd-key.asc
----

. Export the public key and import it into your keyring (needed to encrypt secrets for ArgoCD).
+
[source,console]
----
gpg --armor --export <fingerprint> > argocd-key-pub.asc
----

. Update the `argocd-k8s-235` fingerprint in `.sops.yaml` with the new fingerprint.

. Place the private key in the `deploy-pgp-key.yaml` secret in each ArgoCD overlay. Edit the secret with `sops` and paste the contents of `argocd-key.asc` into the `deploy-key.asc` field.
+
[source,console]
----
sops k8s/apps/argocd/overlays/235-prod/secrets/deploy-pgp-key.yaml
sops k8s/apps/argocd/overlays/305-1700-prod/secrets/deploy-pgp-key.yaml
----

. Re-key all existing k8s secrets so they are encrypted with the new key.
+
[source,console]
----
find k8s/**/secrets/*.yaml -name '*.yaml' | xargs -i sops updatekeys -y {}
----

. Clean up the exported key files.
+
[source,console]
----
rm argocd-key.asc argocd-key-pub.asc
----

=== 🔥 Prometheus

[.text-justify]
Cluster-wide metrics monitoring in my clusters is performed by prometheus. More specifically, I use ArgoCD to deploy https://git.tdude.co/tristan/kube-prometheus-k8s-235/-/tree/master/[my personal configuration] of the https://github.com/prometheus-operator/kube-prometheus/[kube-prometheus] project. Like everything else, it is deployed by ArgoCD and configured to scrape ``ServiceMonitor``s and ``PodMonitor``s in all namespaces, along with cluster-wide internal metrics. Each cluster has its own kube-prometheus deployment with cluster-specific manifests. The web interface for prometheus is reachable at https://prometheus.monitoring.tdude.co and the one for Grafana at https://monitoring.tdude.co. Grafana is configured for SSO authentication with Hashicorp Vault's OIDC provider.

[.text-justify]
Additionally, `kube-prometheus` also deploys alertmanager and a handful of useful alerting rules. I have configured it to notify me on Discord and on Matrix when alerts are firing using https://github.com/benjojo/alertmanager-discord/[alert-manager-discord] and https://github.com/jaywink/matrix-alertmanager/[matrix-alertmanager]. The web interface for alertmanager is available at https://alertmanager.tdude.co.

=== 🛩️ Traefik

[.text-justify]
The idiomatic way to expose http services in kubernetes is via an ingress controller, a fancy word for a reverse proxy that does some service discovery. For this task, I use https://traefik.io/traefik/[Traefik] for no particular reason other than having used it in the past and having liked it.

==== Single Sign-On

[.text-justify]
Traefik supports a host of interesting features like its extensive middleware system. I make extensive use of the "Forward Auth" middleware to protect services I don't want exposed to the public behind an OpenID Connect SSO login via https://github.com/oauth2-proxy/oauth2-proxy/[oauth2-proxy]. The OIDC provider backing oauth2-proxy is Hashicorp Vault's identity OIDC provider. SSO protection can be added to any ingress by setting the `traefik.ingress.kubernetes.io/router.middlewares` label on it to `traefik-forward-auth@kubernetescrd`.

==== TLS

[.text-justify]
Traefik is configured to accept TLS 1.2 and 1.3 connections only and sets strict-transport security headers unconditionally. While it is capable of obtaining TLS certificates on its own via ACME, I choose to disable that functionality and provision TLS certificates with https://cert-manager.io/[cert-manager] instead because it is much more flexible. Setting the `cert-manager.io/cluster-issuer` label to `"letsencrypt-prod"` on any ingress will cause cert-manager to provision a certificate matching the domains on that ingress. Cert-manager is configured to obtain certificate from the Let's Encrypt staging and production environments via the DNS-01 challenge using Cloudflare's API.

=== 💾 Democratic-csi

[.text-justify]
No kubernetes cluster is complete without persistent storage for stateful applications. My hypervisors aren't particularly interesting in terms of storage, but I do have a NAS with plenty of storage so let's use that! I use the https://github.com/democratic-csi/democratic-csi/[democratic-csi] CSI storage driver in my clusters to provide programmatic access to storage for my kubernetes nodes.

[.text-justify]
On the k8s-235 cluster, democratic-csi makes calls to the TrueNAS HTTP API on `spitfire` to provision new ZFS datasets or ZVOLs and exports them over NFS or iSCSI depending on the type of share. The configuration in `k8s/apps/democratic-csi` defines 4 distinct ``StorageClass``es for k8s-235:

- `freenas-nfs-csi`
- `freenas-iscsi-csi`
- `truenas-nfs-spitfire-fast`
- `truenas-iscsi-spitfire-fast`

[.text-justify]
The non-fast storage classes use HDD-backed ZFS pools while the fast ones use NVMe SSD-backed pools. `freenas-nfs-csi` is used for large file collections like media and documents where bulk capacity matters more than latency. `truenas-nfs-spitfire-fast` provisions SSD-backed ZFS datasets over NFS for smaller files that benefit from faster storage. `truenas-iscsi-spitfire-fast` provisions SSD-backed ZFS datasets over iSCSI and is the go-to choice for database workloads that require block-level semantics and proper file locking. `freenas-iscsi-csi` is a legacy HDD-backed iSCSI storage class.

[.text-justify]
On the k8s-305-1700 cluster, democratic-csi provisions ZFS datasets from local storage on `spike` itself rather than from a remote NAS.

==== Configuration

[.text-justify]
To in order for democratic-csi to do its work, we need to supply it with a TrueNAS API key and an SSH private key to log into TrueNAS as root to run some ZFS commands (this might not be required in the future).

.Procedure to add an SSH key to TrueNAS root
[%collapsible]
====
democratic-csi uses SSH to execute ZFS commands on TrueNAS as root. The private key is stored in the SOPS-encrypted driver-config secret files in each democratic-csi overlay.

. Generate an SSH keypair.
+
[source,console]
----
ssh-keygen -t ed25519 -N "" -f democratic-csi-truenas
----

. Add the public key to TrueNAS root. In the TrueNAS WebUI, navigate to Accounts > Users > root and paste the contents of `democratic-csi-truenas.pub` into the SSH Public Key field.

. Update each of the 4 SOPS-encrypted driver-config secret files with the new private key. Edit each file with `sops` and replace the `sshConnection.privateKey` value with the contents of the private key file.
+
[source,console]
----
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-nfs.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-nfs-spitfire-fast.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-iscsi.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-iscsi-spitfire-fast.yaml
----

. Clean up the keypair from the local machine.
+
[source,console]
----
rm democratic-csi-truenas democratic-csi-truenas.pub
----
====

.Procedure to generate and retrieve a TrueNAS API key
[%collapsible]
====
democratic-csi uses the TrueNAS HTTP API to provision ZFS datasets and manage NFS/iSCSI shares. Authentication is done via a Bearer token API key.

. Log into the TrueNAS WebUI at `https://spitfire.235.tdude.co`.

. Navigate to the API Keys settings (gear icon in the top right > API Keys) and click Add to create a new API key. Copy the generated key.

. Update each of the 4 SOPS-encrypted driver-config secret files with the new API key. Edit each file with `sops` and replace the `httpConnection.apiKey` value.
+
[source,console]
----
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-nfs.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-nfs-spitfire-fast.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-iscsi.yaml
sops k8s/apps/democratic-csi/overlays/235-prod/secrets/driver-config-iscsi-spitfire-fast.yaml
----
